# AI Architecture Diagnosis And Plan

Snapshot: 2026-06-07

Scope: diagnosis and implementation plan only. No runtime code, migrations, database
changes, or production prompt changes are included here.

## Executive Summary

The current AI works, but its responsibilities are mixed across prompt, code,
RAG, URA, ticket state, and hardcoded backend decisions.

The target architecture should separate responsibilities:

- Prompt guides behavior.
- Knowledge base provides reliable content.
- Conversation context provides continuity.
- Tools execute real actions.
- Code validates permissions, state, safety, and execution.
- Logs audit decisions and outcomes.
- Human fallback handles unresolved or high-risk cases.

Recommendation: implement by phases, moving toward a professional agent
architecture without breaking the current Whaticket flow.

- `backend/src/services/AiServices/AiTaggerService.ts`

Main models/tables:

- `AiInteractionLogs`
- `KnowledgeBaseArticles`
- `Tickets`
- `Messages`
- `Queues`
- `UraFlows`
- `UraOptions`
- `QueueDistributionLogs`

Current flow:

1. Message arrives in `handleWhatsappEvents.ts`.
2. Ticket, contact, queue, URA, and WhatsApp state are resolved.
3. URA can route or start AI through `START_AI`.
4. AI decision is handled by `DecideAiTicketActionService`.
5. RAG search runs through `SearchKnowledgeBaseService`.
6. Provider call runs through `GenerateAiResponseService`.
7. Backend interprets the returned JSON action.
8. Backend sends message, transfers, closes, or updates ticket state.
9. Logs are written to `AiInteractionLogs` and operational logs.

## 4. Forced Responses And Forced Flows

Existing forced behavior examples:

- Greeting response generated by backend before full AI decision.
- `AI_NO_SAFE_ANSWER_HANDOFF_MESSAGE`.
- Handoff when action is `sem_resposta_segura`.
- Handoff after confirmation attempts exceed limit.
- Closing with fixed close message/tag.
- URA menu responses and queue position messages.
- Fallback queue from URA/AI option.

These should be classified:

- Keep as guardrail: permissions, state transitions, handoff execution, close execution.
- Rework as agent policy: conversational wording, qualification strategy, product/service reasoning.

## 5. Current Conflicts

### Code vs Prompt

Risk:

- Prompt may ask AI to qualify, but backend can turn missing/invalid action into
  handoff.

Observed example:

- Model returned useful text with `response_value`.
- Backend expected `acao`.
- Backend interpreted as `sem_resposta_segura`.
- Ticket went to human.

### Duplicate Rules

Rules appear in both:

- `DecideAiTicketActionService`
- `GenerateAiResponseService`
- `handleWhatsappEvents`
- AI behavior prompt from admin settings

This creates maintenance and behavior drift.

### Hardcoded Conversation Text

Some hardcoded responses should become configurable prompt/agent templates.

## 9. Target Architecture

```text
AI Agent
├── Global prompt
├── Specific agent/fila/URA prompt
├── Scoped knowledge base
├── Current conversation context
├── Structured ticket summary
├── Collected data
├── Missing data
├── Allowed tools
├── Forbidden tools
├── Safety rules
├── Ticket state
├── Recent history
├── Logs
└── Human fallback
```

Responsibility split:

| Layer | Responsibility |
|---|---|
| Prompt global | Universal behavior and limits |
| Prompt specific | Agent role, tone, domain policy |
| Knowledge base | Reliable products/services/procedures/prices |
| Context memory | What this customer already said in this ticket |
| Tools | Real actions with backend execution |
| Code | State, permissions, validation, guardrails |
| Logs | Audit and quality evaluation |
| Human | Fallback and high-risk decisions |

## 10. Prompt Strategy

### Global Prompt

Short, stable, generic:

- Be clear, polite, objective.
- Use current ticket context.
- Use relevant knowledge base.
- Do not invent facts.
- Ask when information is missing.
- Clarify contradictions.
- Do not confirm real actions without successful tool execution.
- Respect allowed tools.
- Escalate only when needed.

### Specific Prompt

Per agent/fila/sector/URA:

- Role
- Business objective
- Tone
- Qualification strategy
- Required data
- Knowledge scope
- Allowed/prohibited actions
- Escalation policy

Example: commercial agent:

- Qualify need.
- Ask before recommending.
- Use price/plan base.
- Recommend best fit.
- Do not close sale without tool/authority.
- Do not confirm schedule without agenda tool.

## 13. Agent Consultivo De Vendas

Behavior:

1. Understand customer need.
2. Ask objective qualifying questions.
3. Use context already collected.
4. Search knowledge base.
5. Recommend best fit.
6. Explain why.
7. Inform price/range only if in base.
8. Prepare preliminary orientation/budget if allowed.
9. Do not confirm sale without tool/authority.
10. Escalate only when needed.

When information is missing:

- Ask one or two targeted questions.
- Do not transfer immediately.

When question is confusing:

- Clarify with a narrow question.
- Do not invent interpretation.

When to transfer:

- Customer asks for human.
- Repeated clarification attempts fail.
- Negotiation depends on human.
- Final price depends on human confirmation.
- Availability needs confirmation and no agenda tool exists.
- Customer wants to close without tool.
- Topic is outside scope.
- Customer is irritated.
- Risk of wrong guidance is high.

## 14. Code As Guardian

Keep in code:

- Authentication and permission.
- Ticket status transitions.
- Queue transfer execution.
- Close execution.
- Tool validation.
- Tool execution.
- Allowed/forbidden tool enforcement.
- Context isolation.
- Prompt injection defense.
- Logs/audit.
- Human fallback execution.

Move out of hardcoded code where possible:

- Business-specific conversation wording.
- Product/service recommendation rules.
- Sector-specific qualification logic.
- Long prompts duplicated across services.
- Automatic step advancement without tool result.

## 15. Guardrails

### Pre-AI Guardrails

- Confirm ticket is eligible for AI.
- Confirm agent/fila/URA scope.
- Load only permitted knowledge base.
- Load current ticket context only.
- Limit recent history.
- Sanitize unnecessary/private data.
- Detect prompt injection attempts.
- Check allowed tools.

### Post-AI Guardrails

- Validate returned JSON/action.
- Block unsupported action.
- Block forbidden tool.
- Block false confirmation.
- Check if response claims scheduled/sold/done without tool success.
- Validate handoff/close tags.
- Validate that values/prices came from allowed base.
- Log decision and blocked reason.

## 21. Options Compared

### Option 1: Keep Architecture And Improve Prompt

Pros:

- Fast.
- Low immediate risk.
- Minimal database work.

Cons:

- Does not solve context memory.
- Does not solve code/prompt conflicts.
- Does not add tool validation.
- Prompt can become huge.

Risk: medium. It may hide architectural problems.

Effort: low.

Recommendation: use only as a temporary patch.

### Option 2: Remove Conversation Rules From Code And Use Prompt + Base

Pros:

- More flexible.
- Less hardcoded behavior.

Cons:

- Unsafe without tool guardrails.
- Higher chance of false confirmations.
- Harder to enforce permissions.

Risk: high.

Effort: medium.

Recommendation: do not do this alone.

### Option 3: Full Professional Agent Architecture

Pros:

- Scalable.
- Auditable.
- Safer.
- Supports multiple sectors/agents.
- Clear separation of responsibilities.

Cons:

- Larger implementation.
- Needs migrations and UI changes.
- Needs careful rollout.

Risk: medium if phased; high if big-bang.

Effort: high.

Recommendation: target state.

### Option 4: Phased Implementation Toward Option 3

Pros:

- Safest.
- Lets current system keep running.
- Delivers value each phase.
- Allows approval gates.

Cons:

- Takes discipline.
- Temporary overlap between old and new architecture.

Risk: low/medium.

Effort: incremental.

Recommendation: best path.

## 22. Recommendation

Use Option 4.

Do not rewrite everything. Build the new architecture in slices:

1. Document and separate current responsibilities.
2. Introduce prompt/global/specific structure.
3. Add structured context memory.
4. Add scoped knowledge base.
5. Add guardrails.
6. Add formal tools.
7. Add logs and quality metrics.
8. Move conversation rules out of hardcoded code gradually.
