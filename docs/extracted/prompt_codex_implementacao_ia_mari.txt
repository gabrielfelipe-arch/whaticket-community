1. DIREÇÃO DE ARQUITETURA
A IA deve funcionar assim:
Cliente conversa
->
Sistema verifica se a mensagem responde à última pergunta pendente
->
Sistema verifica se é opção de menu pós-orçamento
->
Sistema interpreta intenção semântica
->
Sistema busca informação na base/RAG quando necessário
->
Sistema chama ferramenta/backend quando for ação crítica
->
IA gera resposta natural usando contexto confiável
->
Sistema salva estado e evita loop
Regra central:
IA conversa.
Base informa.
Motor de cálculo calcula.
Backend valida.
Atendente resolve exceções.

7. RESOLVEDOR DE PERGUNTA PENDENTE
Criar ou ajustar um serviço antes da IA chamado:
PendingQuestionResolver
Responsabilidade:
Resolver respostas curtas com base no estado da conversa, sem gastar IA quando não precisa.
Exemplos:
Se awaiting.field = participantCount e cliente responde “20”:
salvar participantCount = 20;
perguntar próximo dado faltante ou chamar cálculo.
Se awaiting.field = occurrenceCount e cliente responde “3”:
salvar occurrenceCount = 3.
Se awaiting.field = durationPerOccurrence e cliente responde “4”:
salvar durationPerOccurrence = 4.
Se awaiting.field = menuOption e cliente responde “1”:
executar opção 1 do menu, que é confirmar disponibilidade/reserva com a equipe.
Se awaiting.field = menuOption e cliente responde “2”:
iniciar nova simulação.
Se awaiting.field = menuOption e cliente responde “3”:
perguntar qual é a dúvida ou responder a dúvida caso ela já venha junto.
Regra:
A mesma resposta “3” muda de significado dependendo da última pergunta da IA.
Se o cliente responder uma pergunta nova enquanto havia pergunta pendente, não forçar dado.
Responder a pergunta e depois retomar o orçamento.
Exemplo:
IA perguntou “Para quantas pessoas seria?”
Cliente: “Onde fica?”
Resposta:
“A Salinha fica na Rua Dias da Cruz, 185, sala 215, Méier. Voltando ao orçamento: para quantas pessoas seria?”

9. FERRAMENTAS / BACKEND
Criar ou ajustar uma camada:
AiCriticalActionExecutorService
Responsabilidade:
Executar ou bloquear ações críticas.
Ações críticas:
calculateQuote;
transferToHuman;
closeTicket;
searchKnowledgeBase;
sendPriceTable;
validateCapacity;
handleDiscountRequest;
handleReservationRequest;
handleAvailabilityRequest;
handlePostQuoteMenuOption.
Regras:
calculateQuote:
só chama quando existirem os 3 dados:
- participantCount;
- occurrenceCount;
- durationPerOccurrence.
deve usar o motor oficial.
não aplicar desconto automático.
deve retornar orçamento bruto de tabela.
resposta final deve conter aviso de simulação informativa.
no primeiro orçamento personalizado da simulação atual, incluir resumo dos itens inclusos.
após orçamento, mostrar menu curto com 3 opções.
handlePostQuoteMenuOption:
opção 1: confirmar disponibilidade/reserva com a equipe;
opção 2: iniciar nova simulação;
opção 3: tratar como outra dúvida.
handleDiscountRequest:
não calcula desconto.
não informa percentual.
encaminha para atendente ou responde que equipe avalia.
handleReservationRequest:
informa regra de 50% + 50%.
não confirma reserva.
encaminha para equipe/ferramenta.
handleAvailabilityRequest:
não confirma disponibilidade sem ferramenta.
encaminha para equipe/ferramenta.
transferToHuman:
só dizer que encaminhou se a ferramenta realmente executou.
se não houver ferramenta, responder que pode solicitar apoio da equipe.
closeTicket:
só encerrar quando pedido for claro.
“fechar negócio” não é encerrar ticket.
“quero fechar” significa intenção comercial, não encerramento do atendimento.
10. RAG / BUSCA NA BASE
Se já existir busca na base, revisar e garantir que:
busca por embeddings ou similar seja usada quando a pergunta for sobre fatos da empresa;
perguntas semanticamente diferentes encontrem o trecho correto;
a resposta use apenas trechos recuperados + regras do backend;
se não houver trecho confiável, a IA diga que precisa confirmar com a equipe.
Se não existir RAG, criar uma primeira versão simples:
dividir base em chunks por seção;
salvar título da seção, conteúdo e embedding;
buscar top K trechos relevantes;
passar os trechos para a IA como “contexto autorizado”;
impedir que a IA invente fora do contexto.
Exemplos:
“tem ar?” deve buscar estrutura inclusa.
“cabe 25?” deve buscar capacidade.
“onde fica?” deve buscar endereço.
“divide no cartão?” deve buscar formas de pagamento/reserva.
“tem pacote professor?” deve buscar Pacote Professor Particular.
“manda a tabela” deve buscar tabela oficial de preços.

12. BLOQUEIO DE LOOP
Criar verificação antes de enviar resposta final.
Se a nova resposta for igual ou muito parecida com a última resposta enviada:
bloquear;
incrementar repeatedResponseCount;
gerar resposta alternativa baseada no estado;
se repeatedResponseCount >= 2, reconhecer o loop e fazer pergunta objetiva.
Exemplo:
Cliente diz:
“você está repetindo”
Resposta esperada:
“Você tem razão, me repeti. Vou retomar direto: quer mudar pessoas, dias ou horas do orçamento anterior?”
Não repetir a mesma oferta.
